## Анализ и архитектура «лёгкого процедурного движка» в стиле Farbrausch

Проведя глубокий анализ предоставленных исходных кодов Farbrausch (kkrunchy, V2, OpenKTG, генератор кубов, шейдеры, FRIED и другие), можно выделить ключевые идеи, которые делают их подход уникальным. Наша цель — не слепое копирование, а **творческая реанимация** этих принципов в современном игровом движке, где процедурная генерация, малый размер и высокая производительность идут рука об руку.

Ниже я представлю **структуру и схему** такого движка, разбив его на модули, объяснив, какие исходные компоненты можно использовать, а какие — переосмыслить.

---

## 1. Ключевые компоненты исходного кода и их потенциал

| Компонент | Что даёт | Как использовать в новом движке |
|-----------|----------|--------------------------------|
| **kkrunchy** (exepacker, rangecoder, model_asm) | Адаптивное арифметическое сжатие, дизассемблер x86 | Взять **алгоритм сжатия** (rangecoder + контекстная модель) для упаковки ассетов на этапе сборки. Можно портировать на C++/SIMD. |
| **V2** (synth_core.cpp, v2mplayer, dsio) | Процедурный синтезатор звука по MIDI, речевой синтез (ronan) | Использовать как **звуковой движок** для генерации музыки и эффектов. Патчи занимают байты, а звук — килобайты. |
| **OpenKTG** (gentexture.cpp) | Процедурная генерация текстур (шум, клетки, bump mapping, цветокоррекция) | Основа для **текстурного пайплайна**. Генерировать материалы на лету или при загрузке уровня. |
| **geneffectcubes** | Динамическая генерация тысяч кубов (BounceWorm, Wirbeln) с тенями/отражениями | Алгоритмы **системы частиц** и динамической геометрии. Можно расширить до генерации травы, камней, врагов. |
| **FRIED** | Формат сжатия изображений с DCT + энтропийное кодирование | Использовать как **внутренний формат текстур** (компактный, с поддержкой альфа). |
| **Шейдеры** (colorcorrect, glare) | Пост-эффекты на ассемблере PS/VS 2.0 | Переписать на HLSL/GLSL для современных API. |
| **BSpline, Perlin, PNG loader** | Математические утилиты | Взять готовые реализации (B‑сплайны для анимации, Perlin для шума). |

## 2. Философия и цели проекта

Мы не стремимся к рекордному сжатию (хотя это приятный бонус). Наша цель — **создать движок, где контент рождается из кода и параметров, а не из гигабайтов файлов**. Это даёт:

- **Бесконечную вариативность** — каждый уровень/текстура/звук могут быть уникальными.
- **Мгновенную загрузку** — не нужно читать с диска мегабайты текстур.
- **Малый размер билда** — игра может весить мегабайты, но выглядеть и звучать как современная.
- **Адаптивность** — процедурные параметры могут подстраиваться под железо.

При этом мы используем **современные возможности GPU** (compute shaders, mesh shaders, инстансинг), чтобы сделать процедурную генерацию быстрой.

## 3. Архитектура движка (схема)

Вот предлагаемая **модульная структура**. Каждый модуль можно разрабатывать независимо.

```
┌─────────────────────────────────────────────────────────────┐
│                       Игровой рантайм                        │
│  (геймплей, физика, ИИ, управление)                         │
└───────────────────┬─────────────────────────────────────────┘
                    │
    ┌───────────────┼───────────────────────────────────────┐
    │               │                                       │
    ▼               ▼                                       ▼
┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐
│  Генератор  │ │  Генератор │ │  Генератор │ │  Рендеринг │
│  уровней    │ │  геометрии │ │  текстур   │ │  и пост-   │
│  (процедур- │ │  (частицы, │ │  (OpenKTG  │ │  эффекты   │
│  ный мир)   │ │  меши,     │ │  + FRIED)  │ │  (шейдеры) │
└────────────┘ └────────────┘ └────────────┘ └────────────┘
      │               │               │              │
      └───────────────┼───────────────┼──────────────┘
                      ▼               ▼
              ┌────────────┐   ┌────────────┐
              │  Сжатие     │   │  Звук      │
              │  (kkrunchy  │   │  (V2)      │
              │  + ассеты)  │   │            │
              └────────────┘   └────────────┘
```

### 3.1. Модуль сжатия (Pack)

**Что делает:** упаковывает любые бинарные данные (шейдеры, конфиги, процедурные сиды) в компактный формат, а также сжимает финальный исполняемый файл.

**Что взять из исходников:**
- `rangecoder.cpp` / `.hpp` — арифметический кодер.
- `model_asm.asm` — контекстную модель (переписать на C++ с SSE для скорости).
- `exepacker.cpp` — логику упаковки PE (можно адаптировать под свой формат).

**Современная адаптация:**
- Использовать **LZ4** или **zstd** для быстрого распаковывания, а арифметическое сжатие оставить для маленьких данных (патчи, сиды).
- Сжимать не только EXE, но и **снэпшоты состояния игры** (сохранения) — будет очень компактно.

### 3.2. Модуль звука (Audio)

**Что делает:** синтезирует музыку и звуки по MIDI-подобным командам, используя патчи (наборы параметров).

**Что взять из исходников:**
- `synth_core.cpp` — C++ версию синтезатора (легко портируется).
- `v2mplayer.cpp` / `.h` — плеер для сжатых MIDI-дорожек.
- `ronan.cpp` — речевой синтезатор (фонемы).
- `dsio_cpp.cpp` — DirectSound вывод (можно заменить на OpenAL или SDL).

**Современная адаптация:**
- Добавить **генерацию MIDI-событий в реальном времени** (адаптивная музыка).
- Сделать **API для загрузки патчей из JSON или бинарных blob**.
- Опционально: рендерить звук не только на CPU, но и на **GPU через CUDA** (маловероятно, но возможно).

### 3.3. Модуль текстур (Textures)

**Что делает:** генерирует текстуры (diffuse, normal, specular) из математических описаний (шум, клетки, градиенты) и фильтров.

**Что взять из исходников:**
- `gentexture.cpp` / `.hpp` — все генераторы (Noise, Cells, GlowRect, Blur, Bump и пр.).
- `loadpng.cpp` — для загрузки внешних текстур (если нужно).
- `fried.hpp` / `encode.cpp` / `decode.cpp` — компактный формат хранения сгенерированных текстур.

**Современная адаптация:**
- Перенести **генерацию на compute shader** (DirectCompute / OpenCL). Это даст огромный прирост скорости.
- Использовать **GPU-шейдеры** для генерации текстур по требованию (например, для динамических декалей).
- Оставить CPU-генерацию только для прекомпиляции (загрузочные экраны).

### 3.4. Модуль геометрии и частиц (Geometry)

**Что делает:** создаёт тысячи динамических объектов (кубы, частицы, траву) с физикой и анимацией, рендерит их с тенями и отражениями.

**Что взять из исходников:**
- `geneffectcubes.cpp` — логику BounceWorm, Wirbeln, рендеринг с тенями (shadow map).
- `cubehorde_shader.hpp` (не показан, но подразумевается) — шейдеры для кубов.
- Алгоритмы сортировки по глубине для прозрачности.

**Современная адаптация:**
- Переписать на **инстансинг (instance rendering)** вместо ручных батчей.
- Использовать **compute shaders** для обновления позиций частиц (GPU‑физика).
- Для теней — **cascaded shadow maps** или **virtual shadow maps**.
- Для отражений — **planar reflections** через зеркальную матрицу (как в оригинале) или **ray tracing** (если поддерживается).

### 3.5. Модуль рендеринга и пост-эффектов (Rendering)

**Что делает:** финальная обработка кадра (цветокоррекция, блики, глубина резкости).

**Что взять из исходников:**
- `effect_colorcorrect.vsh` / `.psh` — цветовая матрица.
- `effect_glare.vsh` — множественные выборки для бликов.
- `effect_jpeg.vsh` — простой вершинный шейдер.

**Современная адаптация:**
- Переписать на **HLSL/GLSL** (можно автоматически из ассемблера).
- Интегрировать в **рендер-граф** (Render Graph) для автоматической оптимизации.
- Добавить эффекты **HDR, bloom, motion blur** на основе тех же идей (свёртки, смешивание).

### 3.6. Модуль генерации уровней / миров (World)

**Что делает:** создаёт процедурный ландшафт, комнаты, коридоры, расставляет объекты.

**Что взять из исходников:** нет готового, но есть **B‑сплайны** (`_bsplines.cpp`) для сглаженных траекторий и **Perlin noise** (используется в OpenKTG). Можно адаптировать `Grid` и `Worms` для создания органических структур.

**Реализация с нуля:**
- Использовать **алгоритм wave function collapse** или **маршевые кубы** для генерации пещер.
- Комбинировать с **шумом Перлина** для ландшафта.
- Сохранять уровень в виде сида + параметров (размер несколько байт).

## 4. Схема взаимодействия модулей (data flow)

```
[Запуск игры]
      │
      ▼
[Загрузка сида уровня и параметров]   (несколько килобайт)
      │
      ├─────► [Генерация мира] ──────► (создание геометрии, коллизий)
      │              │
      │              ▼
      │         [Генерация текстур] (OpenKTG на GPU/CPU)
      │              │
      │              ▼
      │         [Генерация звука] (V2, синтез музыки по MIDI-паттернам)
      │              │
      │              ▼
      │         [Сборка сцен] (объединение объектов)
      │
      ▼
[Игровой цикл]
   │
   ├─► [Обновление частиц/кубов] (CPU/GPU)
   ├─► [Рендеринг теней] (shadow map)
   ├─► [Рендеринг отражений] (stencil + зеркало)
   ├─► [Рендеринг непрозрачных объектов]
   ├─► [Рендеринг прозрачных] (сортировка по глубине)
   └─► [Пост-эффекты] (цветокоррекция, блики)
```

## 5. Технологический стек (рекомендации)

| Компонент | Технологии |
|-----------|------------|
| Язык | C++20 (или Rust для безопасности) |
| Рендеринг | Vulkan / DirectX 12 (позволяют compute shaders, инстансинг) |
| Аудио | OpenAL Soft + синтезатор V2 (как статическая библиотека) |
| Физика | CPU (можно свою простую) |
| Скриптинг | Lua или собственный виртуальный стек (как в Werkkzeug) |
| Сборка | CMake + vcpkg / conan |
| Сжатие | Собственный range coder (или zstd) + kkrunchy для финальной упаковки EXE |

## 6. План разработки (поэтапно)

### Этап 1: Базовые процедурные генераторы
- Портировать **OpenKTG** на современный C++ и добавить compute-шейдеры для шума.
- Портировать **V2** в виде отдельной библиотеки (убрать зависимость от ассемблера, оставить C++).
- Создать тестовое приложение, которое генерирует текстуру и звук.

### Этап 2: Геометрическая подсистема
- Реализовать инстансинг кубов (на основе `geneffectcubes`).
- Добавить динамическую генерацию через `BounceWorm` и `Wirbeln` (переписать на CPU + SSE).
- Сделать простую систему теней (один directional light + shadow map).

### Этап 3: Рендеринг и пост-эффекты
- Написать базовый рендерер (Vulkan / DX12) с загрузкой шейдеров.
- Интегрировать пост-эффекты (цветокоррекция, блики).
- Реализовать отражения от плоскости (пол/вода) через stencil.

### Этап 4: Звук и музыка
- Создать MIDI-секвенсор (на основе `v2mplayer`), который генерирует события из процедурных паттернов.
- Добавить речевой синтез (Ronan) для голосов NPC.

### Этап 5: Процедурный мир
- Написать генератор пещер/левелов (wave function collapse + Perlin).
- Научить движок расставлять кубы/частицы по сгенерированной геометрии.

### Этап 6: Упаковка и оптимизация
- Интегрировать `kkrunchy` для сжатия EXE (опционально).
- Сжимать ассеты (сиды, шейдеры) через range coder.
- Оптимизировать производительность (профилирование, SIMD).

## 7. Отличия от исходного подхода Farbrausch

| Аспект | Оригинальный .kkrieger / Werkkzeug | Наш движок |
|--------|-----------------------------------|------------|
| Цель | Минимальный размер (96 КБ) | Процедурность + малый размер |
| Генерация текстур | На CPU при загрузке | На GPU (compute) или при загрузке |
| Звук | V2 на CPU | V2 на CPU + возможность GPU-синтеза |
| Геометрия | Фиксированное количество кубов | Динамическое, инстансинг |
| Тени | Только карта теней 2К | CSM + виртуальные тени |
| Отражения | Плоскость + stencil | Плоскость + stencil (можно добавить SSR) |
| Размер билда | 96 КБ | Несколько МБ (но это не главное) |
| Язык | Ассемблер + C++ | C++20 + шейдеры |

## 8. Пример кода: как будет выглядеть инициализация в новом движке

```cpp
#include "ProcEngine.h"

int main() {
    ProcEngine engine;
    
    // 1. Загрузить сиды (всего несколько байт)
    engine.LoadSeed("level1.seed");
    
    // 2. Сгенерировать мир (процедурно)
    engine.GenerateWorld();   // вызывает генераторы текстур, геометрии, звука
    
    // 3. Запустить игровой цикл
    while (engine.IsRunning()) {
        engine.Update();      // обновление частиц, физики, MIDI
        engine.Render();      // рендеринг с тенями, отражениями, пост-эффектами
    }
    
    return 0;
}
```

Внутри `GenerateWorld()`:
- `TextureGen::GenerateNoise(seed, ...)` → текстура 1024×1024 через compute shader.
- `MeshGen::GenerateTerrain(seed, ...)` → меш ландшафта.
- `AudioGen::SynthesizeMusic(seed, ...)` → заполняет буфер V2 патчами.

## 9. Заключение

Представленный анализ и архитектура показывают, что **идеи Farbrausch живы и могут быть переосмыслены** в современном контексте. Мы не стремимся к 96 КБ, но сохраняем дух процедурной генерации, малого размера и уникального контента.

Ключевые компоненты (kkrunchy, V2, OpenKTG, geneffectcubes) являются **готовыми прототипами**, которые можно взять за основу. Остаётся перевести их на современные рельсы: использовать GPU, SIMD, инстансинг, но не терять ту элегантность и простоту, которая делала демосцену великой.

**Следующий шаг** — создание репозитория с модульной структурой и постепенное портирование. Если потребуется помощь в реализации конкретного модуля (например, портирование OpenKTG на compute shader), я готов детализировать.
