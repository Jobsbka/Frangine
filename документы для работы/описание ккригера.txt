==KKRIEGER==

📁 kkrunchy/
Исполняемый упаковщик (executable packer) для Windows. Используется для сжатия готового исполняемого файла игры до минимального размера. Это собственный инструмент группы Farbrausch, который позволил достичь итоговых 96 КБ.

📁 ktg/
Инструмент генерации текстур OpenKTG. Отвечает за процедурное создание текстур на основе их «истории создания» вместо хранения попиксельных данных.

📁 v2/
Многофункциональный синтезатор, который генерирует музыку и звуки в реальном времени на основе непрерывного потока MIDI-данных. Без него игра была бы беззвучной, а размер аудиоресурсов — огромным.

📁 werkkzeug3_kkrieger/
Основная директория, содержащая само творческое окружение .werkkzeug3 (инструмент для создания демосценных продуктов) и компоненты, специфичные для .kkrieger. Внутри неё находятся следующие подпапки:

Подпапка	Назначение
fried/	Кодек Farbrausch для кодирования/декодирования изображений (Farbrausch Image Encoder/Decoder).
materials/	Система прямого управления материалами (Direct Material System). Используется для определения свойств поверхностей в игре.
player_demo/	Проигрыватель (плеер) для файлов Werkkzeug, предназначенный для демо-версий.
player_intro/	Проигрыватель для интро (вступительных заставок) в формате Werkkzeug.
player_kkrieger/	Проигрыватель, который непосредственно запускает игру .kkrieger из файлов Werkkzeug.
pngloader/	Библиотека для загрузки PNG-изображений (Portable Network Graphics). Содержит заголовочные файлы и саму библиотеку.
shadercompile/	Компилятор шейдеров от Farbrausch. Преобразует код шейдеров либо в бинарный вид, либо в заголовочный файл C/C++.
werkkzeug3/	Ядро программы .werkkzeug3 — инструмент для создания демосценных работ, включая .kkrieger.
📁 yasm-1.3.0/
Ассемблер YASM (версия 1.3.0) — полная переработка NASM под лицензией BSD. Используется для сборки низкоуровневых частей проекта, критичных к производительности и размеру кода.

⚙️ Примечания по сборке
Проект собирается в Visual Studio 2022 Community.

Требуется добавить папку yasm-1.3.0/ в системную переменную PATH перед сборкой.

Весь код распространяется под лицензией BSD или передан в общественное достояние (в зависимости от конкретного компонента).

Таким образом, каждая папка отвечает за свой этап создания игры: от генерации текстур и музыки до упаковки и компиляции шейдеров, а также предоставляет среду разработки (.werkkzeug3) и плееры для запуска результата.



По папкам
📁 1. kkrunchy/ — упаковщик исполняемых файлов
Назначение: сжатие готового EXE до минимального размера (96 КБ для .kkrieger).
Какие файлы из вашего списка сюда относятся:

Файл	Роль
exepacker.cpp / .hpp	Главный класс EXEPacker. Читает PE-файл, фильтрует код через DisFilter, упаковывает через RangecoderPack, линкует depacker через RDFLinker.
depacker.asm	Stage-1 depacker на ассемблере — распаковывает сжатую секцию.
depack2.asm	Stage-2 depacker — восстанавливает исходный код после фильтрации (disfilter).
model_asm.asm	Модель контекстного сжатия (адаптивная арифметика) — вызывается из rangecoder.cpp.
rangecoder.cpp / .hpp	Арифметический кодер/декодер с контекстным моделированием. Вызывает model_asm.
rdf.cpp / .hpp	Парсер и линковщик RDF-объектов (формат, который генерирует YASM). RDFLinker собирает depacker из нескольких объектных файлов.
dis.cpp / .hpp	Дисассемблер x86. Анализирует код, заменяет инструкции на более компактные формы, размечает jump-таблицы.
debuginfo.cpp / .hpp	Обработка отладочной информации (из MAP или PDB). Используется для анализа размера функций в упакованном виде.
mapfile.cpp / .hpp	Чтение .map файлов (символы, секции).
pdbfile.cpp / .hpp	Чтение .pdb файлов через DIA SDK (для VC++).
incdata.asm	Встраивает бинарники depacker.rdf и depack2.bin в исполняемый файл упаковщика.
_startconsole.cpp / .hpp	Обёртка консольного приложения (загрузка файлов, таймеры, вывод).
_types.cpp / .hpp	Базовые типы, строки, математика, массивы, векторы — общая библиотека.
main.cpp	Точка входа sAppMain. Парсит аргументы, вызывает EXEPacker::Pack.
kkrunchy.vcxproj	Проект Visual Studio для сборки утилиты kkrunchy.exe.
Итог: папка kkrunchy/ — это самостоятельный инструмент, который упаковывает любые EXE-файлы, не только .kkrieger. Именно он сжимает игру до 96 КБ.

2. Звуковой синтезатор V2 — процедурная генерация звука
Это полноценный синтезатор, работающий по MIDI-командам. Он генерирует звук в реальном времени, используя всего несколько килобайт патчей. Именно он создаёт музыку и звуки в .kkrieger.

Архитектура (файлы)
Файл	Что делает
synth.asm	Ядро синтезатора на ассемблере: осцилляторы, фильтры, огибающие, LFO, эффекты. Полностью самодостаточно.
synth_core.cpp	C++ реплика того же синтезатора (для читаемости). Идентичен по логике.
v2mplayer.cpp / .h	Плеер для формата .v2m (V2 Music) — читает сжатые MIDI-дорожки и кормит ими синтезатор.
libv2.h	C-интерфейс к синтезатору: synthInit, synthRender, synthProcessMIDI, synthSetGlobals.
sounddef.cpp / .h	Определения параметров патчей (V2 Patch). Фактически — схема памяти для звуковых пресетов.
ronan.cpp	Речевой синтезатор (фонемный). Генерирует речь по тексту.
dsio_cpp.cpp	DirectSound вывод с низкой задержкой, потоковый рендеринг.
scope.cpp	Отладочный осциллограф (OpenGL) — полезен для визуализации сигналов.
Как работает синтезатор
Имеет 16 MIDI-каналов, каждый может загрузить свой патч (инструмент).

Патч — это набор параметров (форма волны, фильтр, огибающая, LFO, модуляции).
Все вычисления — float, 32 бита, с фиксацией денормалов (чтобы не тормозить).
Рендеринг блоками (размер кадра — около 128 сэмплов, зависит от частоты дискретизации).
Эффекты: реверберация, хорус/фленжер, компрессор, дисторшн, эквалайзер.

Применение в вашем движке
Используйте synth_core.cpp как основу для процедурной музыки и звуков. Он не требует внешних библиотек (кроме <math.h>).
Адаптируйте v2mplayer для воспроизведения ваших собственных MIDI-подобных потоков (можно генерировать музыку алгоритмически).
Возьмите ronan.cpp для синтеза речи или вокальных эффектов (фонемы можно менять).
Интегрируйте dsio_cpp.cpp как простой аудиовыход через DirectSound (или перепишите под OpenAL / SDL).
Преимущество V2: он уже процедурный — патчи занимают всего ~200 байт, а звук генерируется по параметрам. Это идеально для вашей концепции «лёгкого движка».

3. Взаимодействие: как .kkrieger использует V2
В игре нет звуковых файлов — вся музыка и звуки синтезируются V2 по MIDI-дорожкам, хранящимся в сжатом виде (формат .v2m).
Плеер v2mplayer читает эти дорожки и в реальном времени отдаёт MIDI-команды синтезатору.
Синтезатор рендерит звук в буфер, который затем выводится через DirectSound.
Это идеальный пример процедурной генерации звука: размер звуковых данных — несколько килобайт, а на выходе — полноценный трек.

2.1. Класс Pixel (структура цвета)
Хранит r, g, b, a как sU16 (0..65535).
Поддерживает:
Инициализацию из sU32 (0xAARRGGBB).
Линейную интерполяцию Lerp.
Композитинг: Add, MulC, ROver, Screen.

2.2. Класс GenTexture — основной рабочий инструмент
Размеры текстур должны быть степенями двойки (удобно для сверток, преобразований).
Генерация из процедурных примитивов
Noise — генерация шума (Perlin или белый шум) с возможностью:
Частота, октавы, затухание.
Абсолютное значение (NoiseAbs).
Нормализация (NoiseNormalize).
Полосовое ограничение (NoiseBandlimit).
Cells — диаграмма Вороного (voronoi) с настраиваемыми центрами и цветами.
GlowRect — «светящийся прямоугольник» (используется для создания сложных узоров, например, в demo.cpp).

Фильтры и преобразования
ColorMatrixTransform — цветовая матрица 4×4 (с опциональным преумножением альфы).
CoordMatrixTransform — аффинное преобразование координат текстуры (масштаб, поворот, сдвиг).
ColorRemap — ремаппинг цвета через одномерные градиенты.
CoordRemap — искажение координат с помощью другой текстуры (например, для эффектов воды).
Derive — вычисление градиента или карты нормалей по высоте (bump map).
Blur — размытие с прямоугольным ядром (использует скользящее окно, эффективно).

Комбинирование текстур
Paste — наложение одной текстуры на другую с множеством режимов смешивания:
Арифметика: Add, Sub, MulC, Min, Max.
Композитинг: Over, Multiply, Screen, Darken, Lighten.
Работа с альфа-каналом: SetAlpha, PreAlpha.
LinearCombine — взвешенная сумма нескольких текстур с константой.
Ternary — условный выбор или смешивание по третьей текстуре.
Освещение и бамп-мэппинг
Bump — освещение поверхности по карте нормалей (с поддержкой направленного или точечного источника, спекулярных карт, карт затухания).

3. Демонстрационный пример (demo.cpp)
В файле demo.cpp показано, как за 100 итераций создаётся текстура металлической пластины:
Генерируются 4 случайные текстуры Вороного с разной плотностью.
Они линейно комбинируются.
Применяется размытие.
Добавляется шумовой слой.
Выполняется цветовая коррекция (Colorize) с помощью матричного преобразования.
Создаётся сетчатый узор (GlowRect) и трансформируется матрицей (поворот, масштабирование).
Из него генерируется карта нормалей (Derive).
Выполняется бамп-мэппинг: базовый цвет + нормали + освещение.
Второй узор накладывается поверх режимом Multiply.
На выходе — готовая текстура, которую можно сохранить в TGA.
Это отличный шаблон для создания собственных процедурных материалов.

4. Как OpenKTG связан с .kkrieger и V2?
В .kkrieger текстуры генерировались процедурно, но код был тесно связан с Werkkzeug3 и не выделен в отдельную библиотеку.

OpenKTG — это более зрелая, документированная и чистая реализация тех же идей.
Вместе с V2 (звук) и kkrunchy (сжатие) OpenKTG даёт три кита для создания полностью процедурного контента:

Звук — синтезируется V2.
Текстуры — генерируются OpenKTG.
Код и ассеты — сжимаются kkrunchy.
