Введение в архитектуру Wz4 / Altona
Проект wz4 (Werkkzeug 4) — это фреймворк для создания интерактивных 3D-демосцен и, вероятно, игр. Он построен поверх библиотеки altona, которая предоставляет низкоуровневые возможности: работа с системой Windows, графикой (DirectX 9/11), вводом, звуком, файлами, сериализацией и базовой математикой.
Ключевые архитектурные решения
1. Многоуровневость:
   * Altona Base: Абстракция над ОС и графическим API (base/graphics.hpp, base/system.hpp). Предоставляет унифицированные интерфейсы для текстур, геометрии, шейдеров, рендер-таргетов, независимые от конкретного sRENDERER.
   * Altona Util: Вспомогательные утилиты: работа с изображениями (image.hpp), сжатие (bitio.hpp, fastcompress.hpp), алгоритмы (algorithms.hpp), парсеры (scanner.hpp), пулы памяти (blobheap.hpp).
   * Wz4 Framework (wz4frlib): Надстройка над Altona, реализующая концепции игрового движка: узлы рендера (wz4_demo2), меши (wz4_mesh), материалы (несколько систем: wz4_mtrl, wz4_modmtrl, wz4_rovemtrl), системы частиц (fxparticle), физика (PhysX), BSP-деревья, воксели (Marching Cubes), шрифты и прочие эффекты.
2. Система рендера на основе графа сцен:
   * Базовый класс Wz4RenderNode и его наследники образуют дерево рендера.
   * Поддержка матриц трансформации (обычные, скиннинг, инстансинг) через флаги (sRF_MATRIX_ONE, sRF_MATRIX_BONE и т.д.).
   * Скриптовые параметры: Классы Wz4RenderAnim... и Wz4ParticlesAnim... используют ScriptContext для анимации параметров через скриптовый язык (вероятно, Lua-подобный или внутренний DSL). Это позволяет анимировать практически любые значения в редакторе.
3. Гибкая система материалов:
   * Wz4Mtrl — базовый класс. Wz4Material (старый стиль), SimpleMtrl, CustomMtrl, ModMtrl (модульный) и RoveMtrl (специализированный).
   * Модульный материал (ModMtrl): Ключевая инновация. Материал собирается из «модулей» (MtrlModule), которые генерируют фрагменты HLSL-кода. Это позволяет динамически создавать шейдеры под нужные фичи (освещение, карты нормалей, тени, туман) без перекомпиляции движка.
   * Constant Buffers: Активно используется для передачи параметров (матрицы, цвета камеры, положение источников света).
4. Работа с геометрией:
   * Wz4Mesh: основное представление полигональной модели. Поддерживает вершины (Wz4MeshVertex), грани (Wz4MeshFace), скелетную анимацию (Wz4Skeleton).
   * ChaosMesh: альтернативное, более «толстое» представление, возможно, для импорта из XSI и последующей оптимизации.
   * Генерация примитивов (куб, сфера, тор) прямо в коде.
   * Разбиение на чанки (Chunks): физические чанки для разрушаемых объектов.
5. Система частиц:
   * Wz4ParticleNode: базовый класс для генераторов. Func(...) вычисляет позиции.
   * RNChunks, RNSprites, RNTrails, RPMarchingCubes — узлы рендера, которые визуализируют частицы как спрайты, объекты, трассы или воксельные облака (Marching Cubes).
   * Интеграция с PhysX для физики частиц (RPPhysxParticleTest, RPPxPart).
6. Объектная модель и сериализация:
   * wObject — базовый класс для всех сущностей редактора (меши, текстуры, материалы, сцены).
   * Собственная система сериализации (sReader/sWriter), использующая операторы | для компактного и безопасного (endian-safe) сохранения/загрузки. Версионирование через Header().
   * Сборщик мусора (sObject): Управление памятью для узлов графа сцен, с корнями (sAddRoot) и маркировкой (Tag()).
7. Низкоуровневый графический менеджмент:
   * sRenderTargetManager_: управление временными рендер-таргетами (кэширование, прокси).
   * sGeometry: буферы вершин/индексов. Поддержка динамической загрузки (BeginLoadVB, EndLoadVB).
   * Поддержка DXT-сжатия текстур, включая кастомные форматы вроде sTEX_DXT5N (для нормалей) и sTEX_DXT5_AYCOCG.
8. Процессинг изображений (Image Post-Processing):
   * Классы IppHelper2 и узлы рендера (RNColorCorrect, RNBlur, RNDof, RNSSAO) реализуют полноэкранные эффекты.
   * Используют sSetTarget для смены рендер-таргетов и рисуют полноэкранные квадраты через sGeometry.
Ключевые паттерны проектирования
* Фабрика (sCreateVertexFormat, sCreateShader).
* Посетитель (сериализация через Serialize_(streamer &s)).
* Компоновщик (Wz4RenderNode и его дочерние узлы).
* Стратегия (разные sMaterial для разных режимов рендера).
* Пулы объектов (sSmallObjectPool, sBlobHeap).
* RAII (широкое использование конструкторов/деструкторов для управления ресурсами DirectX).
* Атомарные операции для потокобезопасности (sAtomicInc).

Раздел 1: Системные основы (Altona Base)
1.1 Абстракция графического API (graphics.hpp/cpp)
Файлы graphics.hpp и graphics.cpp образуют многоуровневую абстракцию над графическим API (Direct3D 9/11, OpenGL, OpenGL ES). Основная идея — скрыть детали реализации за единым интерфейсом, используя макросы (#if sRENDERER == sRENDER_DX11) и общие структуры.
Ключевые элементы:
   * Глобальные состояния:
   * sGFXRendertargetX/Y/Aspect, sGFXViewRect — текущий рендер-таргет и область окна.
   * sPreFlipHook, sPostFlipHook — хуки для вызова до/после смены кадра (используются для профилирования и пост-эффектов).
   * CurrentCBs[], CurrentCBsMask[] — кэширование текущих константных буферов для минимизации вызовов API.
   * Управление экраном:
   * sScreenMode и sScreenInfo — структуры для описания видеорежимов (разрешение, частота, мультисэмплинг, флаги fullscreen).
   * Функции sSetScreenMode, sGetScreenMode, sGetScreenSize — единый интерфейс для смены режима, независимо от платформы.
   * Вершинные форматы (sVertexFormatHandle):
   * Описываются массивом sU32, где каждый элемент кодирует семантику (sVF_POSITION, sVF_NORMAL, sVF_UV0), тип данных (sVF_F3, sVF_C4, sVF_I4) и поток (sVF_STREAM0..sVF_STREAM3).
   * sCreateVertexFormat создаёт или находит существующий формат в хэш-таблице (кеширование). Для DirectX 11 генерирует ID3D11InputLayout на основе временного шейдера, скомпилированного на лету.
   * sVertexFormatBasic, sVertexFormatStandard и другие — предопределённые форматы для распространённых случаев.
   * Геометрия (sGeometry):
   * Управляет вершинными и индексными буферами, поддерживает статическую (sGD_STATIC), кадровую (sGD_FRAME) и потоковую (sGD_STREAM) загрузку.
   * BeginLoadVB / EndLoadVB — асинхронная загрузка вершин. Для sGD_STATIC выделяется временный буфер, который затем копируется в immutable resource. Для sGD_FRAME/sGD_STREAM используются динамические буферы с маппингом.
   * Draw с поддержкой инстансинга и диапазонов индексов.
   * Quadrics: специальный режим для быстрого рисования 2D-примитивов (квадов, гридов) через единый sGeometry с возможностью смешивания разных типов (реализован через макрос sCONFIG_QUADRICS).
   * Текстуры:
   * sTextureBase — базовый класс. sTexture2D, sTextureCube, sTexture3D.
   * Поддерживаются форматы: ARGB8888, DXT1/3/5, ARGB16F/32F, глубина (PCF), структурированные буферы (для Compute Shader).
   * sTextureProxy — класс-прокси, позволяющий перенаправлять текстуру на другой объект (например, для рендер-таргетов, которые меняются каждый кадр).
   * Загрузка через BeginLoad / EndLoad с поддержкой мип-уровней.
   * DXT-компрессия (кодек от Fabian Giesen) — быстрая, с дизерингом и оптимизацией цветов.
   * Шейдеры:
   * sShaderBlob — контейнер для кода шейдера с поддержкой нескольких платформ (HLSL, GLSL) в одном blob. Методы Get / GetAny для извлечения нужной версии.
   * sShader — инкапсулирует созданный шейдер, хранит код, хэш, счётчик ссылок. Кешируется в глобальном списке Shaders.
   * sCreateShaderRaw — создаёт шейдер из сырых данных. Вычисляет хэш (Adler32) и ищет дубликат перед созданием нового.
   * sCBufferBase — константный буфер. Шаблон sCBuffer<Type> автоматически вычисляет маску регистров и настраивает слот.
   * Рендер-таргеты:
   * sTargetSpec и sTargetPara — описание целей рендера (цвет, глубина, куб-фейс, окно).
   * sSetTarget — универсальная функция установки рендер-таргета. Для DX11 настраивает OMSetRenderTargets, RSSetViewports, ClearRenderTargetView и т.д.
   * sCopyTexture — копирование между текстурами с поддержкой фильтрации.
   * Совместимость старого и нового интерфейса через макросы RENDERTARGET_OLD_TO_NEW и RENDERTARGET_NEW_TO_OLD.
   * Вспомогательные функции:
   * sGetGraphicsCaps — получение возможностей железа (форматы текстур, уровень шейдеров, смещения UV/XY для точного тексельного адреса).
   * sDbgPaintWireFrame — глобальное переключение каркасного режима.
   * sEnlargeZBufferRT — увеличение глубинного буфера для offscreen рендера.
1.2 Реализация для DirectX 11 (graphics_dx11.cpp)
Файл содержит конкретную реализацию абстракций для DX11. Ключевые моменты:
   * Инициализация:
   * InitGFX — создаёт IDXGIFactory1, ID3D11Device, ID3D11DeviceContext. Поддерживает мульти-экраны (sSM_MULTISCREEN).
   * Выбирается уровень возможностей (D3D_FEATURE_LEVEL_11_0 → 9_3). Компилятор шейдеров загружается из D3DCompiler_42.dll.
   * Создаются глобальные менеджеры: GeoBufferManager (для динамических вершинных/индексных буферов) и CBufferManager (пул константных буферов).
   * Кэшируются MSAA-уровни через CheckMultisampleQualityLevels.
   * Контекст потока (sGfxThreadContext):
   * Позволяет записывать команды в ID3D11DeviceContext и затем выполнять их в основном потоке (deferred context). Используется для многопоточного рендера.
   * BeginRecording / EndRecording создают командный лист.
   * Управление вершинными/индексными буферами:
   * sGeoBufferManager выделяет блоки памяти (по умолчанию 2 МБ) и динамически распределяет их для sGD_FRAME/sGD_STREAM. Использует ID3D11Buffer с флагом D3D11_USAGE_DYNAMIC и MAP_WRITE_NO_OVERWRITE.
   * sGeometry::BeginLoadVB запрашивает у менеджера отображённый участок, EndLoadVB — освобождает.
   * Для статической геометрии (sGD_STATIC) создаётся D3D11_USAGE_IMMUTABLE буфер через CreateBuffer.
   * Управление константными буферами:
   * sCBufferManager реализует пул константных буферов фиксированных размеров (бин 64 байт). Для запроса размера N округляет до ближайшего бина и берёт буфер из кэша.
   * sCBufferBase::SetRegs отображает буфер и привязывает его к нужному шейдерному этапу (VS, PS, GS, HS, DS, CS) через *SetConstantBuffers.
   * Текстуры и рендер-таргеты:
   * sTexture2D::Create2 создаёт ID3D11Texture2D с учётом флагов (sTEX_RENDERTARGET, sTEX_MSAA, sTEX_DYNAMIC). Для MSAA создаётся дополнительная мультисэмплированная текстура и флаг NeedResolve.
   * sSetTarget настраивает ID3D11RenderTargetView и ID3D11DepthStencilView. Учитывает кубические карты (DXCubeRenderView).
   * sResolveTarget выполняет ResolveSubresource для MSAA-текстур.
   * Окклюзионные запросы:
   * sOccQuery использует ID3D11Query с типом D3D11_QUERY_OCCLUSION. Результаты асинхронно считываются через GetData.
   * Отличительные особенности DX11-реализации:
   * Использование IDXGISwapChain::ResizeBuffers для обработки изменения размера окна.
   * Поддержка sCSBuffer (Compute Shader buffers) — структурированные, raw, с возможностью unordered access.
   * Поддержка sComputeShader для GPU-вычислений (Dispatch).
1.3 Работа с окнами и GDI (windows.cpp/hpp)
Этот модуль обеспечивает взаимодействие с операционной системой (Windows) для оконного интерфейса, 2D-рисования и ввода.
   * Оконные функции:
   * sSetWindowName, sSetWindowSize, sSetWindowPos, sGetWindowMode — обёртки над WinAPI.
   * sSetMousePointer — скрывает/показывает курсор и меняет его форму (стрелка, ожидание, рука и т.д.).
   * sEnableFileDrop — включает перетаскивание файлов через DragAcceptFiles.
   * Буфер обмена: sSetClipboard / sGetClipboard для текста и для сериализованных объектов (используется sWriter/sReader).
   * 2D-рисование (GDI):
   * sRender2DBegin / sRender2DEnd — переключает контекст рисования на окно или offscreen-битмап.
   * sSetColor2D — задаёт цвет для одного из 32 предопределённых слотов (используются HBRUSH и HPEN).
   * sRect2D, sLine2D, sBlit2D, sStretch2D — базовые примитивы.
   * Клиппинг: sClipPush/sClipPop, sClipRect, sClipExclude — работа с HRGN.
   * Шрифты (sFont2D):
   * Загружает шрифт через CreateFontIndirectW, получает метрики (высота, базлиния, ширина символов через GetCharABCWidths).
   * Умеет печатать текст с переносом слов (sF2P_MULTILINE), выравниванием, поддержкой табуляции, курсора, выделения (sPrintInfo).
   * sImage2D — класс для загрузки/отображения изображений через GDI.
   * Диалоги:
   * sSystemOpenFileDialog — вызов стандартных диалогов открытия/сохранения файла через GetOpenFileNameW / GetSaveFileNameW.
   * sSystemMessageDialog — MessageBoxW.
1.4 Система ввода (input2.cpp/hpp)
Input2 — абстрактная система ввода, поддерживающая джойстики, мышь, клавиатуру и кастомные устройства. Ключевая идея — отделение устройства (физический источник) от схемы (логическая привязка к действиям игры).
   * Устройство (sInput2Device):
   * Абстрактный класс, специфичные реализации (например, для Windows) наследуют sInput2DeviceImpl<NUM>, где NUM — количество осей/кнопок.
   * Устройство хранит очередь событий (sLockQueue<Value_>), которые накапливаются в потоке ввода. В OnGameStep они преобразуются в абсолютные (ValuesAbs) и относительные (ValuesRel) значения.
   * Pressed / Released сравнивают текущее и предыдущее абсолютное значение с порогом.
   * Поддерживаются моторы (вибрация) через SetMotor.
   * Схема (sInput2Scheme):
   * Связывает логический идентификатор действия (например, INPUT_JUMP) с одним или несколькими ключами устройств (sInput2Key).
   * Bind добавляет привязку. Можно привязать несколько клавиш к одному действию (например, и клавиша "Пробел", и кнопка A на джойстике).
   * Методы Pressed, Released, Hold, Analog, Relative, Coords, Position и т.д. возвращают агрегированное состояние по всем привязанным ключам.
   * Поддержка ретриггера (SetRetrigger): действие повторно срабатывает с заданной задержкой и интервалом, если клавиша зажата.
   * Маппинг (sInput2Mapping):
   * Управляет привязками для нескольких игроков. Хранит массив sInput2Key* размера maxKeys * maxPlayers.
   * Глобальные функции:
   * sInput2Update — вызывается каждый кадр, обновляет все устройства.
   * sInput2SendEvent — отправляет событие (используется для GUI, например, клавиатурные события).
   * sInput2RegisterDevice / sInput2RemoveDevice — регистрация устройств.
1.5 Файловый ввод-вывод и сериализация (system.cpp/hpp, serialize.cpp/hpp)
   * Файловая система:
   * sFile — абстрактный файл. Реализации: sMemFile (файл в памяти), sGrowMemFile (растущий), sFailsafeFile (запись через временный файл).
   * sFileHandler — механизм плагинов для разных типов файлов (например, архивов). sAddFileHandler / sRemFileHandler.
   * sCreateFile — создаёт файл через зарегистрированные обработчики.
   * Утилиты: sLoadFile, sSaveFile, sLoadText, sSaveTextAnsi, sCopyFile, sRenameFile, sMakeDirAll.
   * Сериализация (sReader/sWriter):
   * Endian-safe, поддерживает версионирование через Header/Footer.
   * sWriter буферизирует вывод, автоматически сбрасывает буфер при заполнении.
   * sReader может работать с отображённым файлом (Map) или с буфером.
   * Object registry: при записи указателей VoidPtr записывается индекс объекта, зарегистрированного через RegisterPtr. При чтении восстанавливается указатель.
   * Шаблонные методы Once, OnceRef, Array, ArrayNew упрощают сериализацию графов объектов.
   * sLoadObject / sSaveObject — типовые обёртки для загрузки/сохранения одного объекта в файл.
1.6 Математическая библиотека (math.cpp/hpp)
Хотя math.cpp в предоставленных файлах пуст, math.hpp (не показан, но подразумевается) содержит основные типы:
   * Векторы: sVector2, sVector30 (3D), sVector31 (3D с однородной координатой), sVector4.
   * Матрицы: sMatrix34 (аффинная 3x4), sMatrix44 (4x4), sMatrix34CM (column-major для шейдеров).
   * Кватернионы: sQuaternion.
   * Геометрические примитивы: sRay, sAABBox, sFrustum.
   * Функции: sFSin, sFCos, sFPow, sFInvSqrt, sPerlin2D, sPerlin3D, преобразования между матрицами и углами Эйлера, интерполяции (sFade), проекции.
1.7 Управление памятью (types2.cpp/hpp, system.cpp)
   * Аллокаторы:
   * sAllocMem / sFreeMem — основные функции с поддержкой тегов (sAMF_*) и профилирования.
   * sMemoryPool — пул для выделения объектов фиксированного размера.
   * sSmallObjectPool / sSmallObjectPoolStatic — пулы для большого количества маленьких объектов (например, узлов сцены).
   * sBlobHeap — менеджер для больших блоков памяти (вершинные/индексные буферы), с поддержкой дефрагментации.
   * Сборщик мусора для sObject:
   * sObject — базовый класс для всех GC-объектов. Содержит флаг NeedFlag и виртуальные методы Tag() (маркировка) и Finalize() (очистка перед удалением).
   * Корни (sAddRoot / sRemRoot) — объекты, которые всегда считаются достижимыми.
   * sCollect устанавливает флаг, по которому в конце кадра вызывается sCollector. Алгоритм: маркировка (проход по корням, вызов Need() у всех дочерних объектов), затем удаление немаркированных объектов (вызов Finalize() и деструктора).
   * sGCLock / sGCUnlock для потокобезопасного доступа.
   * Сообщения (sMessage):
   * Легковесная система сообщений для sObject. Поддерживает отложенную отправку (Post), синхронную (Send), асинхронную (PostASync).
   * sMessageTimer — периодическая отправка сообщения.
   * Прочие контейнеры и утилиты:
   * sArray<T> — динамический массив с автоматическим ростом.
   * sEndlessArray<T> — массив, который растёт при записи за пределы.
   * sPoolString — интернированные строки (глобальный пул).
   * sHashTable — простая хэш-таблица с открытой адресацией.
   * sBitVector — битовый вектор с автоматическим ростом.

Раздел 2: Вспомогательные утилиты (Altona Util)
Этот раздел охватывает набор независимых или слабо связанных с графикой модулей, которые обеспечивают функциональность, необходимую для создания игр и инструментов: обработка изображений, алгоритмы, сжатие, планирование задач, парсинг, растеризация векторов, профилирование и эффекты.
2.1 Работа с изображениями (image.cpp/hpp)
Модуль предоставляет классы для хранения, преобразования и загрузки/сохранения растровых изображений. Ключевая особенность — поддержка широкого спектра форматов (включая сжатые DXT) и интеграция с графической подсистемой через sTexture.
Классы
   * sImage: Представление изображения в оперативной памяти в формате ARGB8888 (32 бита на пиксель, порядок: синий, зелёный, красный, альфа в little-endian). Содержит размеры (SizeX, SizeY) и указатель на данные (Data). Предоставляет методы:
   * Загрузка/сохранение: Load/Save (автоопределение формата по расширению: BMP, TGA, JPG, PNG, PIC).
   * Манипуляции: Copy, Scale, Half (уменьшение в 2 раза с опциональной гамма-коррекцией), Blur (через разделимые X/Y проходы), Outline (генерация обводки для шрифтов), PMAlpha (премультиплицирование альфы), SwapRedBlue, Fill, Glow, Perlin и другие.
   * Работа с палитрой: MonoToAll, AlphaFromLuminance.
   * Блендинг: Blend, HalfTransparentRect.
   * Генерация шрифтовых страниц (CreateFontPage) — использует GDI для рендеринга символов и упаковки их в атлас.
   * sImageData: Хранит изображение в формате, пригодном для прямой загрузки в текстуру (включая сжатые DXT и HDR-форматы). Используется для сериализации и обмена данными с графическим API. Содержит:
   * Формат (Format), размеры, количество мип-уровней (Mipmaps).
   * Сжатые или несжатые данные (Data), палитру (Palette).
   * Init2 — выделяет память под заданный формат и мип-уровни.
   * ConvertFrom / ConvertTo — преобразование между sImage и sImageData с учётом целевого формата.
   * CreateTexture — создаёт sTexture2D или sTextureCube из данных.
   * Поддержка кодеков (sImageCodecType): можно зарегистрировать свой декомпрессор через sSetDecompressHandler.
   * sImageI16: 16-битное полутоновое изображение (один канал). Используется для карт высот, масок и других одноканальных данных.
   * sFloatImage: Изображение с плавающей точкой (RGBA32F). Поддерживает загрузку из HDR (Radiance), тональную компрессию, гамма-коррекцию, альфа-калибровку.
Форматы текстур и сжатие
   * В sTextureFlags перечислены все поддерживаемые форматы: ARGB8888, DXT1/3/5, ARGB16F/32F, Depth/PCF, MRGB8/16 (сжатие HDR с масштабированием) и другие.
   * DXT-компрессия: Реализована в rygdxt namespace (автор Fabian Giesen). Функция sCompressDXTBlock выполняет сжатие блока 4x4 с дизерингом и опциональным альфа-каналом. Алгоритм:
   1. Поиск главной оси методом PCA (через ковариационную матрицу).
   2. Подбор двух опорных цветов (экстремумы).
   3. Квантование до 565 (RGB) и 3/4 бит на пиксель для альфы (DXT5).
   4. Итеративное уточнение цветов для минимизации ошибки.
   * AYCoCg: Формат сжатия цвета с сохранением яркости. Функции sARGBtoAYCoCg и sAYCoCgtoARGB преобразуют ARGB в YCoCg и обратно, что даёт лучшую PSNR при сжатии DXT.
Загрузка и сохранение
   * sImage::Load определяет формат по расширению и вызывает соответствующий загрузчик (BMP, TGA, JPG через stb_image, PIC — внутренний формат Werkkzeug).
   * sImage::Save поддерживает BMP, TGA, PNG (через stb_image_write) и PIC.
   * PIC-формат: Собственный формат Farbrausch. Поддерживает RLE-сжатие, раздельные каналы (RGB + альфа), 8 бит на канал. Код в LoadPIC и SavePIC.
   * Загрузка текстур: sStreamImageAsTexture читает sImageData из потока (сериализованный объект) и создаёт sTextureBase. Поддерживает сжатые данные через декомпрессор.
Вспомогательные функции
   * Heightmap2Normalmap — преобразует карту высот в карту нормалей (использует фильтр Собеля).
   * sMergeNormalMaps — объединяет две карты нормалей в пространстве одной из них.
   * sGenerateMipmaps — генерирует мип-цепочку для sImageData (поддерживает ARGB8888, ARGB32F, MRGB).
   * sClampToARGB8 — конвертирует HDR-изображение в LDR с клэмпингом.
   * sSaveRT — сохраняет текущий рендер-таргет в BMP.
2.2 Алгоритмы (algorithms.hpp)
Заголовочный файл, содержащий шаблонные реализации обобщённых алгоритмов, работающих с диапазонами (ranges). Основные возможности:
   * Определение диапазонов:
   * sArrayRange<T> — обёртка над непрерывным участком памяти (указатель на начало и конец). Поддерживает произвольный доступ.
   * sDListRange<T, Offset> — диапазон для двусвязного списка (бидирекциональный).
   * sAll — функция для получения диапазона из массива, sArray, sDList.
   * sReverse — адаптер, возвращающий диапазон в обратном порядке.
   * Макрос sFOREACH: Единый интерфейс для итерации по любому диапазону. Использует технику "auto any" для обхода ограничений статической типизации.
   * Сортировка:
   * sHeapSort — пирамидальная сортировка с гарантией O(n log n).
   * sIntroSort — интроспективная сортировка (быстрая + пирамидальная при глубокой рекурсии).
   * sInsertionSort — для маленьких массивов.
   * Функции сравнения: sCmpLess, sMemberLess, sMemPtrLess и т.д.
   * Бинарный поиск:
   * sLowerBound, sUpperBound — поиск позиции вставки.
   * sAllLessThan, sAllGreaterThan, sAllEqual — получение поддиапазонов.
   * sFindFirst — поиск первого вхождения.
   * Пирамида (sBinaryHeap):
   * Поддерживает извлечение наибольших элементов (Pop). Используется в sHeapSort и sIntroSort.
2.3 Побитовый ввод-вывод и кодирование Хаффмана (bitio.cpp/hpp)
Модуль для эффективной работы с битовыми потоками, необходимый для сжатия данных (текстур, геометрии) и кодеков.
   * sBitWriter:
   * Пишет биты в буфер или файл. Буферизирует 32-битное слово, записывает байты при переполнении.
   * PutBits(bits, count) — записывает count бит (до 24).
   * При работе с файлом использует промежуточный буфер 4 КБ и сбрасывает его при необходимости.
   * sBitReader:
   * Читает биты из буфера или файла. Аналогично sBitWriter, но с поддержкой refill при чтении.
   * PeekBits(count) / GetBits(count) — просмотр/чтение битов.
   * sLocalBitReader — локальная копия для ускорения в циклах (предотвращает косвенные обращения).
   * Кодирование Хаффмана:
   * sBuildHuffmanCodeLens — строит длины кодов по частотам (алгоритм, гарантирующий максимальную длину maxLen). При превышении maxLen частоты сглаживаются и алгоритм повторяется.
   * sBuildHuffmanCodeValues — назначает коды в соответствии с длинами.
   * sWriteHuffmanCodeLens / sReadHuffmanCodeLens — запись/чтение длин кодов в поток (с использованием разностного кодирования).
   * sFastHuffmanDecoder: Табличный декодер с прямым поиском для кодов длины ≤ 8 бит. Для длинных кодов используется бинарный поиск по MaxCode. Ускоряет декодирование в 2-3 раза.
2.4 Быстрая компрессия LZP (fastcompress.cpp/hpp)
Алгоритм LZP (Lempel-Ziv + Prediction) для очень быстрого сжатия и распаковки (100-200 МБ/с на Core 2). Компрессор работает поблочно (блок по умолчанию 32 КБ) и использует хэш-таблицу для поиска совпадений.
   * sFastLzpCompressor:
   * Инициализирует хэш-таблицу (HashSize=65536) и кольцевой буфер (ChunkSize=32768).
   * Для каждого блока: сжимает с использованием экспоненциального кода Голомба для длин совпадений. Если сжатый размер больше исходного, блок сохраняется как есть (с флагом uncompressed).
   * Поддерживает пошаговый ввод-вывод (StartPiecewise / WritePiecewise / EndPiecewise), что позволяет сжимать данные по частям (например, при сериализации).
   * sFastLzpDecompressor:
   * Читает заголовок блока (2 бита флага + опционально длина), затем разжимает данные.
   * Для сжатых блоков декодирует длины совпадений и литералы, копируя данные из ранее распакованных.
   * Пошаговый ввод-вывод позволяет распаковывать данные на лету.
   * sFastLzpFile: Обёртка над sFile, использующая LZP-сжатие прозрачно. Записывает в файл магическое слово "FastLZP" и размер несжатых данных в заголовке.
2.5 Планировщик задач (taskscheduler.cpp/hpp)
Система для параллельного выполнения задач с использованием кражи задач (work stealing) — эффективна для гетерогенных вычислительных нагрузок (например, рендер частиц, марширующие кубы).
   * sStsManager — центральный диспетчер.
   * Создаёт пул потоков (на основе sGetCPUCount()). Поток 0 — главный (не создаётся отдельно).
   * Управляет пулом sStsWorkload (рабочих нагрузок).
   * Предоставляет методы BeginWorkload, StartWorkload, SyncWorkload, EndWorkload.
   * sStsWorkload — контейнер для набора задач, которые нужно выполнить параллельно.
   * Содержит массив очередей задач для каждого потока (Queues[thread]).
   * NewTask создаёт задачу (sStsTask), которая содержит указатель на функцию (sStsCode), диапазон подзадач (Start/End) и гранулярность.
   * Память под задачи и синхронизацию выделяется из внутреннего пула Mem для уменьшения фрагментации.
   * sStsThread — поток-исполнитель.
   * Основной цикл: ожидает событие, затем пытается выполнить задачу из своей очереди или украсть из чужих.
   * Алгоритм кражи: Ищет поток с наибольшим количеством подзадач (SubTaskCount), затем либо забирает половину задач, либо расщепляет одну задачу (если диапазон >1). Это балансирует нагрузку.
   * Синхронизация: sStsSync — счётчик, который уменьшается по завершении задач. Когда достигает нуля, может запустить продолжение (ContinueTask).
   * sStsPerfMon — профилировщик для STS.
   * Записывает временные метки (через __rdtsc или sGetTimeUS) для каждого события Begin/End.
   * Визуализирует использование потоков в виде цветных полосок поверх рендера.
2.6 Парсеры (scanner.hpp/cpp, json.hpp, rasterizer.hpp/cpp)
Лексический сканер (sScanner)
Класс для разбора текста на токены. Поддерживает C++-подобные комментарии, строки с escape-последовательностями, числа (шестнадцатеричные, двоичные, с плавающей точкой), идентификаторы и пользовательские токены.
   * Настройка: AddToken, AddTokens, DefaultTokens. Токены могут быть как символами (+, -), так и многосимвольными (<<, ==).
   * Режимы: sSF_CPPCOMMENT, sSF_NEWLINE, sSF_ESCAPECODES, sSF_MERGESTRINGS и т.д.
   * Ввод: Из строки (Start) или файла (StartFile). Поддерживает включение файлов (IncludeFile).
   * Использование: После сканирования токена (Scan) проверяются поля Token, ValI, ValF, Name, String. Удобные методы: ScanInt, ScanFloat, ScanString, Match, IfToken.
   * Обработка ошибок: Error — выводит сообщение с позицией (файл, строка).

Регулярные выражения (sRegex)
Класс для сопоставления с образцом, построенный на недетерминированных конечных автоматах (NFA) с явным отслеживанием состояний. Поддерживает основные возможности:
   * Символы и классы: ., \d, \w, \s, [a-z], [^...].
   * Квантификаторы: *, +, ?.
   * Альтернативы: |.
   * Группировка: (...) с захватом подстрок.
   * Якоря: ^ и $.
   * Алгоритм: Строит NFA по регулярному выражению, затем удаляет ε-переходы и мёртвые состояния. При сопоставлении отслеживает все возможные состояния параллельно. Возвращает все возможные совпадения (не только первое). Захваченные группы можно получить через GetGroup.
JSON-парсер (sJSON)
Простой рекурсивный парсер JSON. Преобразует текст в иерархию объектов типа Number, String, Array, Object. Поддерживает атрибуты (специальная обработка поля @attributes для совместимости с PHP-экспортом XML).
Векторный растеризатор (VectorRasterizer в genvector.cpp)
Класс для растеризации контуров, заданных отрезками и кубическими кривыми Безье. Использует алгоритм со сверхдискретизацией по Y (OVERSAMPLE_Y = 16) и активными рёбрами.
   * Принцип: Контур разбивается на рёбра, которые сортируются по Y. Для каждой строки сканирования (с субпиксельным разрешением) вычисляются пересечения, накапливается покрытие в буфере cover. После обработки всех субпиксельных строк накопленное покрытие применяется к целевому пикселю (с усреднением).
   * Поддержка кривых Безье: Рекурсивное деление до тех пор, пока кривая не станет близка к прямой (<=1 пиксель).

2.7 Профилирование и отладка (perfmon.cpp/hpp)
Модуль для сбора и отображения производительности в реальном времени (наложение поверх 3D-сцены). Включается макросом sPERFMON_ENABLED (отключён в sSTRIPPED).
   * Замеры времени:
   * sPerfEnter / sPerfLeave — замеряют время выполнения блока кода. Сохраняют в буфер событий для каждого потока.
   * sPerfSet — отмечает событие без изменения вложенности.
   * sPerfMuteEnter / sPerfMuteLeave — временно отключают замеры (для критических секций).
   * Данные визуализируются в виде цветных полосок (каждый цвет соответствует разной функции/области).
   * Отслеживание значений:
   * sPerfAddValue — добавляет целочисленную переменную для отображения (например, количество полигонов, FPS). Можно группировать.
   * sPerfAddSwitch — добавляет переключатель (например, режим рендера), который можно менять мышью в окне профилировщика.
   * Визуализация:
   * Рисуется поверх сцены через sPainter. Показывает таймлайн потоков (бары), значения переменных и активные переключатели.
   * Управление: стрелки для навигации, Ctrl + клик для изменения значений переключателей.
2.8 Эффекты и анимация (effect.hpp/cpp, animation.hpp/cpp, camera.hpp/cpp)
   * sEffect и sEffectManager_: Простая система управления эффектами. Эффекты имеют методы OnInit, OnFrame, OnPaintPre, OnPaintMain, OnPaintIPP. Менеджер вызывает их в определённом порядке (сортировка по Order). Используется для пост-эффектов, GUI-анимаций и т.п.
   * sFadeData: Класс для плавного затухания/проявления. Поддерживает линейный, плавный, ступенчатый типы. Поля: Time, Length, Dir (IN/OUT). Метод GetFactor возвращает коэффициент (0..1) в зависимости от типа.
   * Камеры (sCamera, sFreeCamera):
   * sCamera — абстрактный класс с виртуальными OnFrame, OnKey. Использует sInput2Scheme для управления.
   * sFreeCamera — реализация свободной камеры: WASD + мышь, регулировка скорости, возможность "летать". Параметры: Speed, LookSpeed, Damping, Pos, DirLook, DirPitch.
2.9 Воксельная графика: Marching Cubes (mcubes.cpp/hpp)
Модуль для генерации полигональной сетки из изоповерхности скалярного поля (например, метаболлов). Содержит:
   * MarchingCubes: Класс, реализующий классический алгоритм Marching Cubes с пространственным хэшированием частиц.
   * Пространственное хэширование: Частицы распределяются по ячейкам сетки (GridSize). Для каждой ячейки, содержащей частицы, вычисляется скалярный потенциал и нормаль в вершинах куба (размер ячейки CellSize). Используется SIMD-ускорение (__m128) для расчёта влияния четырёх частиц одновременно.
   * Рекурсивное уточнение: Если в кубе есть пересечение изоповерхности, но нет вершин с разными знаками, выполняется дополнительное уточнение (субдискретизация) для повышения качества.
   * Геометрический буфер: Результат складывается в GeoBufferHelper, который управляет набором sGeometry для разных потоков.
   * MarchingCubesHelper: Более оптимизированная версия с поддержкой цветных вокселей (MCPotFieldColor).
   * Предварительно вычисляет таблицы для 6 уровней детализации (CellSizeP = 0..5).
   * Использует sStsManager для параллельной обработки ячеек.
   * Генерация вершин и индексов происходит в March<...>.
2.10 Хранилище больших блоков (blobheap.cpp/hpp)
BlobHeap — менеджер памяти для больших блоков данных (например, вершинные буферы, сжатые текстуры). Особенности:
   * Выделяет память крупными кусками (ChunkSize, по умолчанию 128 МБ) и управляет свободными диапазонами через связный список.
   * Потокобезопасен (использует sThreadLock).
   * Alloc возвращает дескриптор sBlobHeapHandle, содержащий массив диапазонов (UseRange), так как блок может быть физически разбит на несколько фрагментов.
   * CopyFrom / CopyInto копируют данные между BlobHeap и обычной памятью с учётом фрагментации.
   * Используется для хранения больших объёмов данных, которые не помещаются в кучу или требуют особого управления.

Раздел 3: Графический движок Wz4
Этот раздел описывает ядро рендеринга — граф сцены, систему материалов, геометрию, частицы и пост-обработку. Все эти компоненты тесно связаны и построены вокруг классов Wz4RenderNode, Wz4Mtrl и Wz4Mesh.
________________


3.1 Граф сцены (wz4_demo2)
Граф сцены — это дерево узлов, каждый из которых может выполнять трансформацию, рендерить геометрию, управлять камерой, накладывать эффекты и т.д. Базовый класс — Wz4RenderNode.
3.1.1 Wz4RenderNode — базовый узел
cpp
class Wz4RenderNode : public Wz4BaseNode {
public:
  sArray<Wz4RenderNode *> Childs;
  sArray<sMatrix34CM> Matrices;  // накопленные матрицы трансформации
  sInt Renderpass;               // для сортировки узлов по pass'у
  ScriptCode *Code;              // скрипт, выполняемый в Simulate
  wOp *Op;                       // оператор, породивший узел (для отладки/перегенерации)
  // ...
};
Ключевые методы (виртуальные, переопределяются в наследниках):
   * Simulate(Wz4RenderContext *ctx) — выполняется один раз за кадр до рендеринга. Обновляет анимированные параметры, вызывает скрипты (SimulateCalc), рекурсивно вызывает Simulate у дочерних узлов (SimulateChilds).
   * Prepare(Wz4RenderContext *ctx) — подготавливает ресурсы (например, обновляет вершинные буферы). Вызывается перед рендерингом, после того как известны все матрицы.
   * Render(Wz4RenderContext *ctx) — отрисовывает узел. Рекурсивно вызывает Render у детей (RenderChilds), если узел не переопределяет поведение.
   * Transform(Wz4RenderContext *ctx, const sMatrix34 &mat) — накапливает матрицы трансформации. Вызывается из родительского узла, передавая текущую матрицу. Добавляет свою матрицу в Matrices и рекурсивно вызывает Transform у детей.
Контекст рендера (Wz4RenderContext) содержит:
   * Текущий sViewport.
   * Скриптовый контекст (ScriptContext).
   * Флаги рендера (RenderFlags), например, wRF_RenderMain, wRF_RenderZ.
   * Время (GetTime, GetBaseTime, GetLocalTime).
   * Текстуры для теней и Z-буфера (ZTarget, ZTargetLowRes).
   * Матрица встряски камеры (ShakeCamMatrix).
Механизм рекурсивных флагов (RecMask): используется, чтобы избежать многократной обработки одного узла при нескольких проходах (например, сначала Z-буфер, потом цвет). Каждый проход увеличивает маску, и узел помечает себя как обработанный.
3.1.2 Конкретные узлы
   * Камера (RNCamera):
   * Устанавливает ctx->View. Поддерживает несколько режимов: target/position, Euler-углы, pivot-камера, сплайн.
   * Управляет очисткой экрана (Clear), зумом, near/far plane.
   * Может рендерить в Rendertarget2D (текстура-цель).
   * Если в сцене есть несколько камер, первая активная задаёт ctx->View, остальные игнорируются.
   * Трансформации:
   * RNTransform — добавляет локальную SRF-трансформацию (Scale/Rotate/Translate).
   * RNTransformPivot — трансформация относительно заданной точки.
   * RNMultiply — повторяет дочерние узлы несколько раз, умножая матрицу (инстансинг).
   * RNLookAt — поворачивает узел, чтобы смотреть на цель.
   * Клиппинг и переключение:
   * RNClip — включает/выключает дочерние узлы в зависимости от глобального или локального времени. Поддерживает петли.
   * RNClipRand — случайным образом выбирает один из дочерних узлов.
   * RNMultiClip — одновременно активирует несколько клипов по таймлайну.
   * Скрипты и переменные:
   * RNAdd — просто вызывает SimulateChilds. Используется как контейнер.
   * RNFader / RNVariable / RNSpline — связывают значения с глобальными символами скриптового движка, делая их доступными для анимации.
   * Рендер геометрии:
   * RNRenderMesh — рендерит Wz4Mesh с заданным материалом и скелетной анимацией.
   * RNMultiplyMesh — создаёт сетку инстансов (матрицы вычисляются как произведение трансформаций).
   * RNSkyCube — рендерит кубическую карту как скайбокс (сдвигает матрицу так, чтобы камера была в центре).
   * 2D-слои:
   * RNLayer2D — рендерит текстуру на весь экран с поддержкой альфа-смешивания, UV-трансформаций, смешивания двух текстур.
   * Анимация по сплайну:
   * RNSplinedObject — перемещает объект вдоль сплайна, вычисляя ориентацию по касательной.
   * RNBoneTrain — анимирует скелет, проецируя кости на сплайн.
3.1.3 Скриптовые анимации
Многие параметры узлов (например, Para.Trans, Para.Scale) помечены как anim в ops-файлах. Для них генерируются классы Wz4RenderAnim..., которые содержат ScriptSymbol*. В Simulate вызывается Anim.Bind(ctx->Script, &Para), что связывает символы с соответствующими полями структуры Para. Скрипт может менять эти значения, и изменения автоматически применяются. Это позволяет анимировать практически любые параметры без написания кода.

3.2 Система материалов
В wz4 существует несколько систем материалов, отражающих эволюцию движка. Основная современная — модульные материалы (ModMtrl), но также есть SimpleMtrl (для простых случаев) и Wz4Material (устаревшая).
3.2.1 Модульные материалы (ModMtrl)
Материал собирается из модулей (MtrlModule), каждый из которых отвечает за определённую функцию: освещение, текстуру, нормал-маппинг, туман, тени и т.д. Модули генерируют фрагменты HLSL-кода, которые затем компилируются в шейдер.
Ключевые классы:
   * MtrlModule — базовый. Методы VS(ShaderCreator *) и PS(ShaderCreator *) добавляют код в вершинный или пиксельный шейдер. Phase определяет порядок выполнения.
   * ModMtrl — сам материал. Содержит список модулей (ModulesUser), а также KillLight, KillShadow, KillFeatures — маски, отключающие ненужные возможности (например, если материал не должен получать тени).
   * ShaderCreator — генератор шейдерного кода. Управляет временными переменными, зависимостями между фрагментами, связывает uniform-переменные с константными буферами.
   * ModEnvNum — окружение (источники света, параметры тумана, clip-плоскости). Для каждого EnvNum (0..15) хранится отдельный экземпляр.
Процесс создания шейдера:
   1. ModMtrl::Prepare() сортирует модули по фазе, вызывает Start у каждого.
   2. Для заданного сочетания флагов (CachedModInfo) ищется готовый шейдер в кэше. Если нет — создаётся новый.
   3. ShaderCreator инициализируется, проходится по модулям, вызывая VS и PS. Модули добавляют код через sc->TB.PrintF(...) и регистрируют зависимости (FragRead, FragModify).
   4. ShaderCreator анализирует зависимости, строит DAG, генерирует итоговый HLSL-код.
   5. Код компилируется через sShaderCompileDX, создаётся sShader.
   6. Создаётся sVertexFormatHandle на основе требуемых атрибутов (позиция, нормаль, тангент, UV и т.д.).
   7. Шейдер и формат сохраняются в CachedModShader.
Пример модуля — текстурный семплер (MM_Tex2D):
   * В Get(ShaderCreator *) возвращает имя переменной, в которой хранится результат сэмплирования.
   * Генерирует код для UV-координат (может брать из атрибутов uv0, uv1, из позиции модели/мира/камеры, из отражённого луча).
   * Поддерживает трансформацию UV (масштаб, поворот, матрица из ModEnvNum).
Пример модуля — освещение (MM_Lambert):
   * В PS генерирует цикл по источникам света (информация о светах приходит из ModEnvNum).
   * Использует ws_lightdir и lighti, вычисляет dot(normal, lightDir), умножает на цвет света, накапливает в col_light.
Константные буферы:
   * ModShaderVEnv (VS) и ModShaderPEnv (PS) — содержат все параметры окружения (матрицы, позиция камеры, цвета).
   * Wz4MtrlModelPara — матрица модели (передаётся из узла рендера).
   * Для скиннинга используется Wz4MtrlSkinPara (массив матриц костей).
3.2.2 SimpleMtrl — упрощённый материал
Использует фиксированные шейдеры (не генерирует код). Поддерживает до 3 текстур, диффузное освещение, туман, clip-плоскости. Настройки передаются через SimpleShaderVPara и SimpleShaderPEnv. Шейдеры SimpleShader имеют пермутации (Tex1, Tex2, Fog, Light), которые выбираются при подготовке.
3.2.3 Устаревшая система (Wz4Material)
Использует предопределённые шейдеры Wz4XShader, Wz4LShader, Wz4CShader. Поддерживает нормал-маппинг, детейл-текстуры, кубические карты, тени. Выбор пермутации происходит на основе набора текстур и флагов.

3.3 Геометрия (wz4_mesh)
Класс Wz4Mesh представляет полигональную модель. Он хранит вершины (Wz4MeshVertex), грани (Wz4MeshFace), кластеры (Wz4MeshCluster) и опционально скелет (Wz4Skeleton) или физические чанки (Wz4ChunkPhysics).
Вершина (Wz4MeshVertex) содержит:
   * Позицию (Pos), нормаль (Normal), тангент (Tangent), бинормаль (вычисляется).
   * Координаты текстур (U0,V0, U1,V1).
   * Веса и индексы костей (Index[4], Weight[4]).
   * Выделение (Select) для редактора.
Грань (Wz4MeshFace) — треугольник или четырёхугольник. Хранит индексы вершин, номер кластера, флаг выделения.
Кластер (Wz4MeshCluster) — группа граней, использующих один материал. Содержит подготовленные к рендерингу sGeometry (обычный и для каркаса), список используемых костей (Matrices), bounding box.
Скелет (Wz4Skeleton) — иерархия костей (Wz4AnimJoint), каждая с каналом анимации (Wz4Channel). Поддерживаются каналы:
   * Wz4ChannelConstant — постоянное значение.
   * Wz4ChannelLinear — линейная интерполяция с фиксированной скоростью.
   * Wz4ChannelPerFrame — покадровая анимация (ключи).
   * Wz4ChannelSpline — B-сплайн интерполяция.
Рендеринг:
   * BeforeFrame — обновляет скелетные матрицы, готовит sGeometry для каждого кластера.
   * Render — выбирает нужный sGeometry, устанавливает константные буферы (матрицы камеры, UV-трансформации), передаёт матрицы костей через sSetVSParam, вызывает Draw.
   * RenderInst — для инстансинга (матрицы передаются в потоке 1).
   * RenderBone — для скиннинга (каждая вершина имеет до 4 весов).
Операции над мешем (реализованы в wz4_mesh_ops.ops):
   * Генерация (куб, сфера, тор, текст, путь).
   * Трансформации (перемещение, поворот, масштабирование, бенд, экструзия).
   * Топологические (сабдивид, фасетирование, сглаживание, разделение по плоскости).
   * Анимация (bake, сжатие каналов в сплайны).
   * Физика (разбиение на чанки, вычисление инерции).

3.4 Система частиц (fxparticle)
Система частиц разделена на генераторы (Wz4ParticleNode) и рендеры (Wz4RenderNode, которые работают с частицами). Генераторы вычисляют позиции и время жизни частиц; рендеры визуализируют их как спрайты, 3D-модели, трассы или воксельные поверхности.
Базовый интерфейс генератора (Wz4ParticleNode):
   * GetPartCount() — количество частиц.
   * GetPartFlags() — какие данные доступны (позиция, ориентация, цвет).
   * Func(Wz4PartInfo &pinfo, sF32 time, sF32 dt) — заполняет массивы pinfo.Parts[] позициями и временами жизни.
Конкретные генераторы:
   * RPBallistic — баллистические частицы (парабола с гравитацией).
   * RPLissajous — движение по фигурам Лиссажу.
   * RPSplinedParticles — движение вдоль сплайна.
   * RPFromMesh — генерирует частицы на поверхности меша (скалярное поле).
   * RPAdd — объединяет несколько генераторов.
   * RPWobble — добавляет шум Перлина к траекториям.
Рендеры частиц:
   * RNSprites — спрайты (квады, всегда повёрнутые к камере). Поддерживает атласы текстур, анимацию, фейд.
   * RNChunks — каждый "чанк" (частица) рендерится как отдельный меш (например, осколок).
   * RNTrails — создаёт ленту (trail) из предыдущих позиций частиц.
   * RNMetaballs — рендерит метаболлы (изоповерхность) через несколько проходов с использованием текстур.
   * RNMarchingCubes — рендерит облако частиц как полигональную поверхность, используя алгоритм marching cubes. Оптимизирован через пространственное хеширование и многопоточность.
Пример: RNMarchingCubes:
   1. Частицы распределяются по хэш-таблице (3D-сетка с ячейками размера Para.GridSize).
   2. Для каждой ячейки, содержащей частицы, вычисляется потенциал и нормали в вершинах куба (func с SIMD).
   3. Применяется алгоритм marching cubes, генерируются вершины и индексы.
   4. Геометрия загружается в sGeometry и рендерится через MarchingCubesHelper.

3.5 Пост-эффекты (IPP)
Пост-эффекты реализованы как узлы рендера, которые читают текущий экран (через sRTMan->ReadScreen()), обрабатывают его и записывают обратно (через sRTMan->WriteScreen()). Для работы используются полноэкранные квады и специальные шейдеры.
Базовый хелпер IppHelper2:
   * DrawQuad — рисует полноэкранный квад с заданными текстурами.
   * Blur — размытие с использованием нескольких проходов (сначала BlurFirst с ядром 4x4, затем BlurNext для увеличения радиуса). Для качества используются два этапа (горизонтальный и вертикальный blur).
Конкретные эффекты:
   * RNColorCorrect — цветокоррекция (умножение, сложение, гамма).
   * RNBlur — размытие с возможностью последующего шарпенинга (unsharp mask).
   * RNDof / RNDof2 — глубина резкости. Генерирует несколько размытых копий (full, half, quarter), затем смешивает их на основе Z-буфера.
   * RNSSAO — ambient occlusion (алгоритм SSAO). Использует случайные направления, семплирует Z-буфер, размывает результат.
   * RNBloom — выделяет яркие области, размывает их и складывает с оригиналом.
   * RNGlow — свечение на основе градиента и маски.
   * RNEdgesDetect — детектор границ (Собель).
   * RNSharpen — повышение резкости.
Шейдеры пост-эффектов написаны в wz4_ipp_shader.asc. Они используют общий вершинный шейдер (проекция) и разнообразные пиксельные шейдеры с выбором пермутаций.

   * Модульные материалы — динамическая генерация шейдеров, что позволяет легко комбинировать функции без перекомпиляции движка.

Раздел 4: Дополнительные подсистемы
4.1 BSP-деревья и построение полиэдров (wz4_bsp.cpp/hpp)
Этот модуль реализует построение BSP-дерева (Binary Space Partitioning) из полигональной сетки и последующую генерацию набора выпуклых полиэдров (чанков). Используется для динамического разрушения объектов (разрезание меша на куски) и, возможно, для collision detection.
Ключевые классы и функции:
   * Wz4BSPFace – грань BSP-дерева.
   * Хранит массив точек (вершин) и плоскость грани (sVector4 Plane).
   * Split – разрезает грань произвольной плоскостью, возвращая две новых грани (front и back). Реализован классический алгоритм: классификация вершин относительно плоскости, добавление точек пересечения рёбер.
   * CleanUp – удаляет дубликаты вершин и вырожденные грани.
   * CalcFaceIntegrals – вычисляет интегралы для физических свойств (объём, центр масс, тензор инерции) по алгоритму Mirtich.
   * Wz4BSPPolyhedron – выпуклый полиэдр, представленный набором граней.
   * InitBBox – создаёт полиэдр в форме ограничивающего параллелепипеда.
   * Split – разрезает полиэдр плоскостью, возвращая два новых полиэдра (передний и задний). При необходимости генерирует новые грани по плоскости разреза для замыкания полиэдров.
   * GenMesh – преобразует полиэдр в Wz4Mesh (экспорт в обычную геометрию).
   * CalcMassProperties – вычисляет объём, центр масс и тензор инерции (используется для физики чанков).
   * Wz4BSPNode – узел BSP-дерева.
   * Может быть Inner (внутренний, с плоскостью и двумя потомками), Solid (сплошной) или Empty (пустой).
   * Хранит ограничивающий AABB (Bounds) для ускорения отсечения.
   * Wz4BSP – основное BSP-дерево.
   * FromMesh – строит BSP-дерево из Wz4Mesh. Алгоритм:
   1. Разбить все грани меша на триангулированные Wz4BSPFace.
   2. Рекурсивно выбрать разделяющую плоскость (по эвристике: минимизировать количество пересекаемых граней).
   3. Разрезать все грани этой плоскостью, распределить по front/back поддеревьям.
   4. Продолжить для поддеревьев.
   * MakeRandomSplits – модифицирует дерево, добавляя случайные разрезы для получения мелких чанков. Использует вероятностный подход, основанный на объёме полиэдра и параметрах minEdge/maxEdge.
   * GeneratePolyhedrons – рекурсивно обходит BSP-дерево, для каждого твёрдого (Solid) листа строит Wz4BSPPolyhedron путём пересечения полупространств (отсечение всеми плоскостями на пути от корня). Результат – набор выпуклых многогранников (чанков).
   * IsInside – проверяет, находится ли точка внутри твёрдого объёма, определённого BSP-деревом.
Применение: Используется в операторе Wz4BSP SliceAndDice для разрезания меша на множество мелких выпуклых чанков, которые затем могут быть использованы в физике разрушений (WpxRigidBodyDebris).

4.2 Воксельная графика (Marching Cubes, fxparticle_mc.cpp/hpp)
Реализует алгоритм Marching Cubes для генерации полигональной сетки из скалярного поля (например, плотности частиц). Используется для визуализации метаболов (облаков частиц) или изоповерхностей в процедурной геометрии.
Ключевые классы и шаблоны:
   * RNMarchingCubesBase<T> – шаблонный класс узла рендера, где T определяет тип данных поля (с цветом или без). Специализации: RNMarchingCubes (только позиция) и RNMarchingCubesColor (позиция + цвет).
   * Пространственное хеширование (Spatial()): разбивает пространство на воксельную сетку (Grid) и для каждой ячейки собирает список влияющих частиц (соседние ячейки). Использует хэш-таблицу для быстрого доступа к ячейкам.
   * Расчёт поля (RenderT): для каждой ячейки вычисляет значение скалярного поля и градиента (нормали) в узлах сетки. Использует SIMD-интринзики (SSE) для ускорения суммирования вкладов частиц.
   * Субдискретизация (опционально): если включена, выполняет дополнительную оценку поля в серединах граней и центре ячейки для сглаживания.
   * Маршинг кубов (MarchingCubesHelper): генерирует вершины и индексы по таблице triTableBase. Поддерживает разные разрешения сетки (от 1x1x1 до 32x32x32).
   * Потоковая обработка: использует sStsManager для распараллеливания обработки ячеек.
   * MarchingCubesHelper – вспомогательный класс для генерации сетки.
   * Содержит предвычисленные таблицы MyTriTable для разных размеров ячейки.
   * Шаблонный метод March выполняет маршинг кубов для заданного блока потенциального поля.
   * GeoBufferHelper управляет пулом вершинных/индексных буферов для многопоточного рендера.
   * Оператор Wz4Render MarchingCubes (в fxparticle_ops.ops) создаёт RNMarchingCubes и связывает его с источником частиц (Wz4ParticleNode).
Применение: Визуализация плотных облаков частиц как непрерывной поверхности (например, дыма, жидкости, энергетических полей).

4.3 Адаптивные поля расстояний (ADF, adf.cpp/hpp)
Модуль Adaptive Distance Fields – сжатое представление знакового поля расстояния (SDF) с использованием октодерева. Позволяет выполнять трассировку лучей и рендерить изоповерхности с высоким качеством при умеренном потреблении памяти.
Ключевые классы (из фрагментов adf.cpp и adf.hpp):
   * tSDF (определён в tADF.hpp, не показан, но используется) – само поле расстояний. Содержит равномерную 3D-текстуру с предвычисленными значениями SDF. Размеры DimX, DimY, DimZ, масштаб и смещение (STBX, STBY, STBZ, OP, MP).
   * GetDistance – интерполирует значение в точке (используется в шейдерах).
   * GetNormal – вычисляет нормаль численным дифференцированием.
   * Wz4ADF – обёртка для tSDF.
   * FromMesh – строит ADF из меша. Алгоритм:
   1. Строит tAABBoxOctree (октодерево с AABB) из меша.
   2. Инициализирует tSDF с заданной глубиной и размером сетки.
   3. Для каждой ячейки октодерева вычисляет знаковое расстояние до ближайшей грани.
   * TraceRay – трассирует луч через поле, используя шаговый алгоритм с адаптивным шагом (на основе расстояния до поверхности).
   * GetTexture – возвращает 3D-текстуру со значениями SDF (формат sTEX_R32F).
   * RNRenderADF – узел рендера ADF.
   * В Render:
   1. Рендерит «градиентную текстуру» (tADFMat), которая для каждого пикселя экрана вычисляет точку пересечения с поверхностью и нормаль, сохраняя их в рендер-таргет.
   2. Затем рендерит итоговое изображение (tADFShadowMat), используя предвычисленную текстуру с нормалями для освещения (модель Фонга) и теней (трассировка к источнику света внутри поля).
   * Шейдеры (adf_shader.asc) выполняют трассировку лучей на GPU, интерполируя SDF из 3D-текстуры.
Применение: Рендер статических и анимированных объектов с гладкой геометрией, создание эффектов размытия, деформаций, морфинга, где высокое качество важнее производительности.

4.4 Шрифты и 2D-текст (chaos_font.cpp/hpp, chaos_font_ops.ops)
Реализует создание текстурных атласов шрифтов и их рендер в 3D-мире или на экране.
Класс ChaosFont:
   * Генерация атласа (InitFont, Letter, Finish):
   * Использует sFont2D из windows.hpp для рендеринга глифов в offscreen-изображение (sImage).
   * InitFont задаёт параметры шрифта (имя, размер, стиль, безопасные отступы).

   * Symbol позволяет добавлять в атлас произвольные изображения (например, иконки).

   * PreparePrint создаёт текстуру атласа, материал (sSimpleMaterial с blend mode sMB_PMALPHA для предварительного умножения альфа) и геометрию.

   * Поддерживает отложенную печать: Print (добавляет в буфер), PrintC (центрирование), PrintR (выравнивание по ширине), Rect (заливка), Paint (отрисовка всего буфера).
Оператор ChaosFont Font (chaos_font_ops.ops) позволяет создавать шрифты прямо в редакторе: задать имя шрифта Windows, набор символов (диапазоны), размер, стиль, отступы.
Применение: GUI в 3D-играх, текстовые эффекты (летающие цифры), отладочная информация.

4.5 Физика (PhysX, wz4_physx.cpp/hpp, wz4_physx_ops.ops)
Модуль интегрирует библиотеку NVIDIA PhysX (версия 3.3.1) в движок. Позволяет создавать симуляции твёрдых тел, джойнтов, частиц.
Архитектура интеграции:
   * Глобальная инициализация (PhysXInitEngine):
   * Создаёт PxFoundation, PxPhysics, PxCooking.
   * Подключает Visual Debugger (PVD) для отладки.
   * Коллайдеры (WpxCollider, WpxColliderBase):
   * Иерархическая система: WpxCollider (примитив: box, sphere, plane, convex hull, mesh) может быть скомбинирован через WpxColliderAdd, трансформирован через WpxColliderTransform или размножен через WpxColliderMul.
   * WpxCollider::CreateGeometry строит визуальное представление коллайдера (меш для отладки).
   * CreatePhysxCollider создаёт PxGeometry и PxShape, прикрепляет к PxRigidActor.
   * Актёры (тела) (WpxRigidBody, WpxRigidBodyDebris):
   * WpxRigidBody – одиночное тело (статическое, динамическое, кинематическое). Управляет массой, инерцией, скоростью, силами.
   * WpxRigidBodyDebris – специальный актёр для разрушаемых объектов. Принимает Wz4Mesh, разбитый на чанки (см. BSP), и создаёт отдельное физическое тело для каждого чанка.
   * PhysxBuildActor создаёт PxRigidStatic или PxRigidDynamic и добавляет в сцену.
   * GetPositionsFromMeshVertices / GetPositionsFromMeshChunks используются для создания множества копий тела в заданных позициях.
   * Джойнты (WpxRigidBodyJoint, WpxRigidBodyJointsChained):
   * Поддерживаются типы: fixed, spherical, revolute, prismatic, distance.
   * WpxRigidBodyJoint соединяет два разных актёра. WpxRigidBodyJointsChained создаёт цепочку соединений между последовательными актёрами (например, для позвоночника).
   * Настройки: лимиты, демпфирование, пружины, брейк-силы.
   * Сцена физики (RNPhysx):
   * Узел рендера, который владеет PxScene. Создаёт сцену с заданной гравитацией, диспетчерами CPU/GPU.
   * В Simulate вызывает scene->simulate() и fetchResults(). Поддерживает синхронизацию с реальным временем (time stepping с фиксированным шагом).
   * Управляет перезапуском симуляции при изменении таймлайна (F6).
   * Физические частицы (RPPhysxParticleTest, RPPxPart):
   * Используют PxParticleSystem (PhysX Particles).
   * Генерируют позиции частиц на CPU, загружают в PhysX, затем PhysX симулирует их (столкновения с коллайдерами сцены).
   * В Func считывают обновлённые позиции и передают их в систему рендера частиц (например, RNSprites).
Операторы (ops) предоставляют GUI для создания коллайдеров, тел, джойнтов и сцены PhysX в редакторе.
Применение: Реалистичная физика объектов (падение, столкновения), разрушаемые конструкции, симуляция жидкостей/пыли через частицы.

4.6 Звук и музыка (sound.cpp/hpp, ogg.cpp/hpp, musicplayer.cpp/hpp)

4.7 Инструментарий отладки (perfmon.cpp/hpp)
Система профилирования производительности (sPerfMon). Позволяет измерять время выполнения участков кода, отображать графики загрузки потоков, отслеживать значения переменных.

   * В конце кадра события визуализируются в виде цветных полосок (время выполнения).
   * Отслеживание значений:
   * sPerfAddValue – регистрирует целочисленную переменную для отображения (имя, диапазон, группа).
   * Значения выводятся в виде числовых индикаторов и гистограмм.
   * Переключатели:
   * sPerfAddSwitch – добавляет отладочный переключатель (например, "off|on"), связанный с целочисленной переменной.
   * Позволяет менять поведение кода в рантайме.
   * Отображение:
   * Рисуется в стиле 2D GUI поверх игрового изображения.
   * sSendPerfMonInput – обрабатывает ввод (стрелки, клики) для навигации по профилировщику.
Применение: Оптимизация производительности, поиск узких мест, отладка многопоточности, интерактивная настройка параметров игры.

Раздел 5: Ключевые аспекты реализации и дизайна
5.1 Потокобезопасность
Проект активно использует многопоточность для повышения производительности, особенно в рендере и задачах (планировщик задач). Основные механизмы синхронизации:
   * Атомарные операции: Для счётчиков ссылок (sAtomicInc/sAtomicDec), флагов и указателей. Реализованы через встроенные функции компилятора (_InterlockedIncrement для MSVC, __sync_add_and_fetch для GCC). Также используются барьеры памяти (sWriteBarrier, sReadBarrier).
   * Спин-блокировки: sStsLock — нерекурсивный мьютекс с активным ожиданием (спином) и последующим вызовом sSleep после определённого числа итераций. Используется в планировщике задач для минимизации накладных расходов.
   * Обычные мьютексы: sThreadLock — обёртка над критической секцией Windows или pthread_mutex_t для Linux. Применяется для защиты не слишком критичных участков (например, очереди сообщений, доступа к пулам ресурсов).
   * Блокировки с областью видимости: sScopeLock — RAII-обёртка для sThreadLock.
   * TLS (Thread Local Storage): sAllocTls выделяет память в контексте потока (sThreadContext). Используется для хранения указателя на текущий контекст (sGetThreadContext), а также для буферов отладки и профилирования.
   * Lock-free очереди: sLocklessQueue<T,max> — кольцевой буфер для одного производителя и одного потребителя. Используется, например, в системе частиц (хотя в предоставленных файлах явно не показана, но sLockQueue — блокирующая версия).
   * Планировщик задач (sStsManager): Реализует work stealing для балансировки нагрузки между потоками. Каждый поток имеет свою очередь задач. При отсутствии задач в своей очереди поток пытается украсть задачи из других очередей (или подзадачи из активной задачи). Это требует осторожной синхронизации (спин-блокировки на очередях) и атомарных операций для счётчиков задач.

5.2 Управление ресурсами
Проект использует комбинацию ручного управления (счётчики ссылок) и автоматического (сборщик мусора для графа сцены).
   * Счётчики ссылок: Классы sShader, sTextureBase, sGeometry, sMaterial и все узлы графа сцены (Wz4RenderNode) наследуют или содержат поле UseCount/RefCount. Методы AddRef/Release управляют временем жизни. Это типичный паттерн для ресурсов, разделяемых между несколькими объектами (например, текстура используется несколькими материалами).
   * Сборщик мусора (sObject): Используется для объектов, образующих граф (сцена, ноды, материалы). GC работает по алгоритму mark-and-sweep:
   * Корни (sAddRoot) — обычно это корневые узлы сцены.
   * Маркировка: sCollect вызывает Need() у корней, который рекурсивно маркирует все достижимые объекты через переопределённый метод Tag().
   * Финализация: вызывается Finalize() для помеченных на удаление объектов (позволяет освободить неуправляемые ресурсы).
   * Удаление: объекты с NeedFlag == 0 удаляются.
   * Пулы объектов:
   * sSmallObjectPool<T> и sSmallObjectPoolStatic<T> — для часто создаваемых/удаляемых объектов (например, узлы BSP-дерева, записи в хэш-таблицах).
   * sBlobHeap — для больших блоков (вершинные/индексные буферы). Управляет фрагментацией, поддерживает выделение и освобождение.
   * Memory Pools: sMemoryPool — простой линейный аллокатор с блоками. Используется, например, в sStringPool_ для хранения интернированных строк.
   * Обработка ошибок:
   * sFatal — безусловное завершение с сообщением.
   * sVERIFY — проверка условия, вызов sFatal при нарушении.
   * sLog и sLogF — логирование с уровнями.
   * sSetErrorCode — установка кода ошибки для командной строки.

5.3 Оптимизации
   * Кэш вершинного буфера: Функция sOptimizeIndexOrder (алгоритм Тома Форсайта) переупорядочивает индексы для улучшения использования post-vertex cache (обычно 24 элемента). Это уменьшает количество вершин, обрабатываемых вершинным шейдером.
   * SIMD (SSE):
   * В модуле Marching Cubes (fxparticle_mc.cpp) функции func используют SSE-интринзики (__m128, _mm_add_ps, _mm_mul_ps, _mm_rcp_ss) для вычисления потенциала и нормали сразу для 4 частиц.
   * В genvector.cpp для растеризации используются SSE-инструкции для ускорения фильтрации.
   * Быстрые математические функции:
   * sFSin, sFCos, sFInvSqrt — оптимизированные версии с использованием таблиц и алгоритма «быстрого обратного квадратного корня».
   * FastEulerXYZ, FastSin в fxparticle.cpp используют предвычисленные таблицы синусов для ускорения тригонометрии в системах частиц.
   * Константные буферы:
   * Кэширование текущих CB (CurrentCBs) и их масок (CurrentCBsMask) позволяет избежать избыточных вызовов VSSetConstantBuffers и PSSetConstantBuffers. При установке нового буфера проверяется, не пересекаются ли регистры с уже установленными, и при необходимости выполняется сброс.
   * Планировщик задач: Разбиение задач на мелкие подзадачи (гранулярность) и динамическое перераспределение (кража) позволяет эффективно использовать многоядерные процессоры. Это особенно важно для Marching Cubes и рендера большого количества объектов.
   * Упаковка вершинных атрибутов:
   * Использование sVF_I4 для нормалей (8 бит на компоненту) вместо sVF_F3 (32 бита на компоненту) уменьшает объём вершинных буферов. В шейдере значения распаковываются: normalize(normal / 127.0 - 1.0).
   * Аналогично, веса костей хранятся как sVF_I4 (unsigned byte) и преобразуются в шейдере.
   * Быстрая компрессия DXT: Кодек rygdxt использует предвычисленные таблицы, дизеринг и оптимизацию цветов (principal component analysis) для достижения высокого качества и скорости.
   * Инстансинг: Широко используется для рендера множества одинаковых объектов (частицы, чанки) через sRF_MATRIX_INSTANCE и sRF_MATRIX_INSTPLUS. Матрицы передаются в виде вершинного атрибута в отдельном потоке.
5.4 Рендер пайплайн
   * Основные фазы:
   * sRF_TARGET_ZONLY — запись глубины (используется для теней, раннего Z-отсечения).
   * sRF_TARGET_ZNORMAL — запись глубины и нормалей (для SSAO, пост-эффектов).
   * sRF_TARGET_MAIN — основной рендер с освещением и текстурами.
   * sRF_TARGET_DIST — запись расстояния (для тумана, DOF).
   * sRF_TARGET_WIRE — каркасный режим.
   * Поддержка теней:
   * Направленные источники: Ортогональная проекция, ограниченная bounding box'ом сцены. Теневые карты с PCF-фильтрацией (4x4 или случайный диск).
   * Точечные источники: Кубические карты теней (6 ортогональных проекций). Используется PxTextureCube (в DX11 через TextureCube).
   * Прожекторы: Перспективная проекция, наложение теневой карты.
   * Baked shadows: Предварительно рассчитанные тени, сохраняемые в текстуру (класс MM_BakedShadow).
   * Пост-эффекты (IPP):
   * Все пост-эффекты реализованы как Wz4RenderNode, которые рендерятся после основной сцены.
   * Используют sRTMan->ReadScreen() для захвата текущего экрана и sRTMan->WriteScreen() для записи результата.
   * Примеры: RNColorCorrect (цветокоррекция), RNBlur (размытие), RNDof (глубина резкости), RNSSAO (screen-space ambient occlusion), RNBloom, RNGlow, RNSharpen.
   * Мультисамплинг (MSAA):
   * Рендер-таргеты могут быть мультисэмплированными (sTEX_MSAA). При смене таргета или перед использованием текстуры как шейдерного ресурса выполняется ResolveSubresource (метод ResolvePrivate).
   * Для Z-буфера разрешение может не выполняться (sTEX_NORESOLVE).
   * Скриптовые шейдеры: RNCustomIPP компилирует HLSL-код на лету и создаёт материал. Это позволяет писать кастомные пост-эффекты без перекомпиляции движка.

5.5 Интеграция с Assimp
В Wz4Mesh есть условная компиляция с Assimp (#ifdef sCOMPIL_ASSIMP).
   * Загрузка: Wz4Mesh::WaiLoadAssimp загружает модель через Assimp с заданными флагами пост-процессинга (например, aiProcess_Triangulate, aiProcess_GenNormals, aiProcess_OptimizeMeshes).
   * Анимация: Assimp-скелет преобразуется во внутреннее представление Wz4Skeleton. Класс Wz4Skeleton содержит специфичные для Assimp поля (WaiRootNode, WaiAnimations, WaiBoneMap).
   * Оценка анимации: Wz4Skeleton::WaiEvaluateAssimpCM рекурсивно обходит узлы Assimp, интерполирует ключи позиции/поворота/масштаба и вычисляет матрицы для каждого кости.
   * Трансформация скелета: Оператор AiTransformBones позволяет трансформировать скелет после загрузки (применяет дополнительную матрицу ко всем костям).
   * Ограничения: Assimp используется только для импорта; в рантайме анимация воспроизводится через собственную систему Wz4Channel (перфреймовая или сплайновая). Это сделано для уменьшения зависимости от Assimp в финальном продукте.
